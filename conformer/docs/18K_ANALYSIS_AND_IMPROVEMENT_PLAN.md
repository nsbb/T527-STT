## 2. 에러 유형 분석

### Insertion / Deletion / Substitution 분포

| 에러 유형 | 전체 | 비율 | 설명 |
|----------|------|------|------|
| **Substitution** | 44,132 | 63.7% | 글자 치환 (가장 많음) |
| **Insertion** | 16,391 | 23.6% | 불필요한 글자 추가 |
| **Deletion** | 8,808 | 12.7% | 글자 누락 |

**핵심 발견: Insertion이 Deletion의 1.86배.** 모델이 글자를 빠뜨리는 것보다 **불필요한 글자를 추가하는 경향이 훨씬 강함.**

### Insertion 과다 패턴 (간투어/환청)

모델이 발화 끝에 불필요한 음절을 추가하는 패턴이 일관적으로 관찰:

```
ref: "관리사무소 전화번호 알려 줘"
hyp: "관리사무소 전화번호 알려 줘 음"     ← "음" 추가

ref: "관리사무소 전화번호가 어떻게 되지"
hyp: "관리사무소 전화번호가 어떻게 되지 어"  ← "어" 추가

ref: "소리 줄여줘"
hyp: "소리 줄여 줘 네"                   ← "네" 추가
```

**흔히 추가되는 토큰:** "음", "어", "네", "에", "아", "응", "둘", "으"

**contains_ref 분석:** CER > 0인 에러 샘플 중 **~80%가 contains_ref=1** — 정답 텍스트는 출력에 포함되어 있으나 뒤에 garbage가 붙어서 CER이 올라가는 것.

### Substitution 패턴

- 음소적으로 유사한 글자 치환: "밝" ↔ "작", "열어" ↔ "연어"
- 자음/모음 혼동: "ㅂ" ↔ "ㅍ", "ㅓ" ↔ "ㅔ"
- 경음화/격음화 혼동: "살려줘" → "살짝 좀"

## 3. 짧은 발화 문제 (가장 심각)

ref_len < 5글자인 짧은 발화에서 CER이 **3.1배 악화:**

| 데이터셋 | 짧은 발화 CER | 긴 발화 CER | 악화 배율 |
|---------|-------------|-----------|---------|
| 010.회의음성 | 46.55% | 14.99% | **3.10x** |
| 009.한국어_강의 | 39.19% | 15.37% | **2.55x** |
| eval_other | 29.17% | 12.43% | **2.35x** |
| eval_clean | 29.05% | 13.43% | **2.16x** |
| 012.상담음성 | 20.50% | 9.90% | **2.07x** |

**원인:**
1. CTC alignment는 충분한 음성 프레임이 있어야 정확 — 짧은 발화는 프레임이 적어 alignment 실패
2. 3초 윈도우에서 짧은 발화(1~2초)는 나머지 1~2초가 silence → blank 토큰 과다
3. 짧은 발화 뒤의 silence를 모델이 추가 토큰으로 해석 (간투어 삽입 원인)

**실패 예시:**
```
ref: "열어" (2글자)
hyp: "열어약해 줘" (5글자, CER 150%)

ref: "네 맞습니다" (5글자)
hyp: "플레반스" (완전히 다른 출력)
```

## 6. 개선 방안 (8가지, 우선순위 순)

### 개선 1: [HIGH] 후처리 — 간투어/반복 토큰 제거

| 항목 | 내용 |
|------|------|
| 문제 | 발화 끝 불필요한 토큰 삽입 ("음", "어" 등) |
| 방법 | CTC decode 후 규칙 기반 필터링 |
| 기대 효과 | **CER 1~2%p 개선** |
| 소요 시간 | 1일 |
| 난이도 | 하 |

구현 규칙:
- 발화 끝 단독 음절 ("음","어","네","에","아","응","둘","으") 자동 제거
- 연속 반복 음절 제거 ("면면" → "면", "이이" → "이")
- 마침표/구두점 뒤 추가 토큰 제거

### 개선 2: [HIGH] QAT (AIHub 데이터)

| 항목 | 내용 |
|------|------|
| 문제 | 양자화 손실 0.66%p + Zeroth QAT 도메인 불일치 |
| 방법 | AIHub 원본 데이터로 FakeQuantize + MarginLoss |
| 기대 효과 | **CER 0.5~1.0%p 개선** |
| 소요 시간 | 4시간 (학습) + 2시간 (변환) |
| 상태 | 서버에서 진행 중 |

### 개선 3: [HIGH] Calibration 확대 (10개 → 100개+)

| 항목 | 내용 |
|------|------|
| 문제 | 10개 calibration으로는 양자화 range 최적화 부족 |
| 방법 | 다양한 도메인에서 100개+ NeMo mel 생성 후 재양자화 |
| 기대 효과 | **CER 0.2~0.4%p 개선** |
| 소요 시간 | 2시간 |
| 난이도 | 하 |

### 개선 4: [MEDIUM] Beam Search CTC 디코딩

| 항목 | 내용 |
|------|------|
| 문제 | Greedy argmax는 차선 경로 무시 |
| 방법 | beam width 10~20으로 CTC beam search |
| 기대 효과 | **CER 0.3~0.8%p 개선** |
| 소요 시간 | 2~3일 |
| 비고 | 기존 NPU output 재활용 가능 (재추론 불필요) |

### 개선 5: [MEDIUM] Language Model (n-gram) 결합

| 항목 | 내용 |
|------|------|
| 문제 | CTC 출력에 문법/맥락 반영 없음 |
| 방법 | KenLM n-gram + beam search fusion |
| 기대 효과 | **CER 1.0~2.0%p 개선** (가장 큰 단일 개선) |
| 소요 시간 | 2~4주 |
| 근거 | SungBeom README: kenlm 추가 시 WER 13→5% |

### 개선 6: [MEDIUM] 입력 길이 확장 (3초 → 5초)

| 항목 | 내용 |
|------|------|
| 문제 | 슬라이딩 윈도우 경계에서 단어 잘림/반복 |
| 방법 | ONNX [1,80,501] → Acuity → 새 NB |
| 기대 효과 | 경계 에러 감소 + 추론 20% 단축 |
| 리스크 | NB 크기 ~130MB → T527 한계 초과 가능 |
| 소요 시간 | 1주 |

### 개선 7: [LOW] 도메인 특화 Fine-tune

| 항목 | 내용 |
|------|------|
| 문제 | 월패드 명령어에 특화되지 않음 |
| 방법 | 월패드 데이터로 LoRA fine-tune |
| 기대 효과 | 월패드 CER 2~5%p 개선 |
| 소요 시간 | 3~4주 |

### 개선 8: [LOW] 슬라이딩 윈도우 오버랩 튜닝

| 항목 | 내용 |
|------|------|
| 문제 | stride 250/301 (17% overlap) — 경계 잘림 |
| 방법 | overlap 33%로 확대 + 겹치는 구간 logit 평균 |
| 기대 효과 | CER 0.1~0.3%p |
| 소요 시간 | 1일 |

## 7. 종합 개선 시나리오

### 단기 (1주 이내)
```
현재:                        CER 15.59%
+ 후처리 (간투어 제거):       → 14.0% (-1.5%p)
+ Calibration 확대 (100개):  → 13.7% (-0.3%p)
+ QAT (AIHub):               → 13.0% (-0.7%p)
```

### 중기 (1개월)
```
+ Beam search (width=20):    → 12.5% (-0.5%p)
+ LM (n-gram KenLM):         → 11.0% (-1.5%p)
```

### 장기 (3개월)
```
+ 5초 NB:                    → 10.5% (-0.5%p)
+ 도메인 fine-tune:           → 9.0% (-1.5%p)
```

**최종 목표: CER 15.59% → 9~11% (4~6%p 개선)**

## 8. 참고: FP32 vs uint8 비교

| | CER |
|---|---|
| FP32 ONNX (Zeroth 100) | 9.93% |
| uint8 T527 (Zeroth 100) | 10.59% |
| **양자화 손실** | **0.66%p** |

양자화 손실이 매우 작아서 (0.66%p), **모델 자체의 FP32 성능(~10%)이 ceiling.** 양자화 개선보다 후처리/LM/디코딩 개선이 더 효과적.
